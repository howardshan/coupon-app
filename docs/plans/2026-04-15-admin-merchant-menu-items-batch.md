## 一、背景与目标

### 1.1 背景

为降低商家入驻与维护成本，希望在**后台管理端**由管理员协助商家**批量维护「商品库」**（与 Deal 解耦的数据）：支持按图片批量创建/更新、表格导入价格、列表内编辑。商家可通过「导出的带 id 的表格」填写价格后，由管理员上传回写。

### 1.2 目标

1. 在**商家管理相关页面**中提供进入「该商家的商品管理」的入口。  
2. 支持**批量创建**：上传多图 → 以**图片文件名**推导商品名（经规范化后匹配现有 `menu_items`）。  
3. **唯一性**：同一 `merchant_id` 下**不允许**两条记录「规范化名」相同；若批量上传时命中已有名称 → **仅更新图片**并**提示**管理员。  
4. **未定价=草稿**：`price` 允许 `NULL`；在列表/导出中显式显示「未定价」；**未定价商品不对外/不在选品中视为可上架**（与现有 deal 选品、用户端展示规则对齐，见后文实现约束）。  
5. **价格维护**：在管理端列表内**直接编辑**（仅管理员）+ **导出 UTF-8 BOM 表格**（id + 名称 + 价格）+ **仅按 `id` 更新**的导入。  
6. 图片走统一 Storage 路径，大批量时采用**异步/进度**等体验优化（可分期）。

## 二、名词与边界

| 名词 | 含义 |
| ---- | ---- |
| 商品 / 商品库 | 表 `public.menu_items` 一行 = 一商品，绑定 `merchant_id` |
| Deal / 团购 | 表 `deals`，本功能**不创建、不审核**；仅保证「有价的商品」才参与选品/上架类逻辑 |
| 规范化名 | 用于**匹配与唯一性**：`trim` 后、**去扩展名**、**大小写敏感** 与库内已存名比较（与上传时从文件名得到的名字使用同一套规范化函数） |

## 三、已确认业务规则（基线）

以下与产品沟通结论一致，开发以此为准；若有变更需更新本章节并记变更记录。

1. **同名**：同商家下**禁止**两行的「规范化名」相同。批量再次命中 → **仅换 `image_url`**，并 **UI 提示**已替换图。  
2. **新商品条件**：规范化后的文件名与库内**无任一条**规范化名完全相等 → 插入新行（系统生成**唯一 `id`（UUID）**）。  
3. **一图一商品**：一个上传文件对应一个 `menu_items` 行（一图一 `image_url`）。  
4. **价格**：`price` 可为 `NULL`；**未定价 = 草稿**，在列表/导出中展示「未定价」；**未定价不允许**作为「可上架/可选用」的终端状态（在查询层或 `status` 扩展中统一，见技术方案）。  
5. **生效与审核**：在后台**直接落库**；不经过 Deal 审核流。  
6. **权限**：管理端以 **Service Role 或服务端受控的 admin 写接口** 为宜，**禁止**在浏览器中暴露能写任意商家数据的密钥。  
7. **存储路径**：`menu-items/{merchant_id}/{item_id}.{ext}`（`ext` 依实际上传类型，不强制写死为 `.jpg`；若需统一可服务端转图，二期考虑）。  
8. **导入导出**：UTF-8 **BOM** 便于 Excel；**导入仅按 `id` 更新**（不依赖名称主键做导入行匹配）。列顺序、表头在实现时**固定**并在界面附「模板下载」。  
9. **换图时旧文件**：应删除或覆盖 Storage 上旧图策略（避免桶内垃圾堆积），在实现单中写清。

## 四、现状与数据模型

### 4.1 当前 `menu_items` 结构（摘自既有迁移）

主要字段包括：`id`、`merchant_id`、`name`、`image_url`、`price`（可空）、`category`、`recommendation_count`、`is_signature`、`sort_order`、`created_at` 等。  

### 4.2 与「草稿 / 未定价」相关的缺口

当前表**未必**有独立 `is_draft` 列。推荐二选一（在 DB 子任务中拍板）：

- **A（推荐、改动小）**：以 **`price IS NULL` 为草稿**；所有 C 端、选品、列表「可上架」的查询**统一**增加 `price IS NOT NULL`（或封装 View / RPC）。  
- **B**：增加 `is_active` 或 `status` 列，与 `price` 共同约束。  

**依赖**：需盘点 **deal_joy** / **dealjoy_merchant** 中**引用** `menu_items` 的页面（如门店详情、创建 Deal 时选品），保证**不能选未定价商品**、用户端不展示无价格行（与产品「未定价不上架」一致）。

### 4.3 唯一性约束

- 建议新增**生成列**或**应用层规范化** + 数据库**唯一约束**：`(merchant_id, name_normalized)` 或等效。  
- `name` 在界面展示用户可读字符串；`name_normalized` 仅用于等值匹配与防重。

### 4.4 RLS

- 管理端写操作经 **Service Role** 时绕过 RLS；或新增 **仅 admin 角色** 的 Policy。需与现网安全模型一致。

## 六、技术方案概要

### 6.1 后端（Supabase）

- **Migration**：`name_normalized`（或等价）、`UNIQUE (merchant_id, name_normalized)`；如采用生成列，规则与代码内 `normalizeMenuItemName()` **一致**。  
- **Storage**：新 bucket 或复用现网 bucket + 目录前缀 `menu-items/...`；**RLS** 或仅服务端用 Service Key 上传。  
- **Edge / Server Action**：在 `admin` 中通过 **Server Action** 或 **Route Handler** 调用 `getServiceRoleClient()` 完成写库+上传。  

### 6.2 前端（admin）

- 新路由：例如 `app/(dashboard)/merchants/[id]/menu/page.tsx`（Server 拉列表 + Client 表单项 / 上传）。  
- 上传使用 `<input type="file" multiple />` 或现网组件；结果 toast + 换图/新建汇总。  
- 导入用 `papaparse` 或手解析 CSV，**必须**处理 BOM。  

### 6.3 与 Flutter/用户端（可选子计划）

- 在「商品对外可见」的查询中过滤 `price IS NULL`。  
- Deal 选品 `menu_item_id` 只展示有价商品。

## 八、验收标准（MVP）

1. 管理员在指定商家下可打开展示该商家全部 `menu_items` 的页面。  
2. 多图上传：同规范化名**仅换图**并有明确提示；新名**插入**且**带 UUID**。  
3. 同商家**无法**因并发创建出两条同规范化名（被 DB 唯一约束或事务挡住）。  
4. `price` 为空时列表/导出**明确**未定价；在约定路径下**不可**作为可上架/可选用。  
5. 导出/导入带 BOM；导入**只按 id** 更新，且**不能**改到其它 `merchant_id` 的商品。  
6. 新图路径符合 `menu-items/{merchant_id}/{item_id}.{ext}` 约定。
