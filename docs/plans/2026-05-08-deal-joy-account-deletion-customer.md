# 客户端（Crunchy Plum / deal_joy）账户删除功能开发计划书

> **文档版本**：2026-05-09  
> **关联文档**：[商家端账户删除计划书](./2026-05-08-account-deletion-merchant-app.md) — 同一阶段**并行**落地；共用 Supabase **Edge Function**（如 `account-delete`）中的用户级分支。  
> **背景**：Guideline **5.1.1(v)** 要求支持账户创建的应用提供账户删除；与商家端一并补齐可降低拒审风险。  
> **代码基线**：`deal_joy/lib/`、`deal_joy/supabase/`。

---

## 1. 目标与验收标准

### 1.1 合规目标（Apple）

- 应用在 **Profile（或合规区域）** 提供可发现的 **Delete account**（或等价英文）入口。  
- 完整删除流程可在 App 内发起并完成（或到达合规的网页完成页且 App 内提供**直达链接**）。  
- 与商家端一致：提审附 **真机录屏** + **Review Notes** 说明。

### 1.2 业务目标

- **整账号删除**：删除 `auth.users`（及计划内的 `public.users` 清理）后，用户无法再使用 **客户端 + 商家端**（同一 Auth）。删除前文案须说明与商家端 **§3.1（整账号）/ §3.2（仅商家身份仅在商家端可选）** 的关系，避免用户误以为仅删客户端。  
- **订单 / 券 / 支付记录**：与商家端已确认策略一致 — **匿名化保留**（见 §5）。  
- **Gift / 转赠券 / 未核销券**：按 §4 规则实现，避免资金与客诉风险。

### 1.3 Definition of Done（客户端）

| 检查项 | 说明 |
|--------|------|
| Profile 内删除入口 | 英文 UI；含不可恢复提示 |
| 调用统一后端 | 与商家端同一 Edge Function 的 **`full`（整账号删除）**；客户端不设「仅消费者」独立 scope，避免与商家 A/B 语义冲突 |
| 敏感模块 | 涉及 `features/auth/`、`payment_methods` 等 **COMPLETED 受保护文件** 时须事先授权 |
| 审核材料 | Review Notes + 录屏（登录 → Profile → Delete → 确认完成） |

## 3. 与商家端的关系

| 场景 | 行为 |
|------|------|
| 用户仅在客户端发起 **整账号删除** | Edge Function `full`：清理消费者数据 + **若存在商家身份则一并按商家计划处理**（或由同一事务编排调用商家分支），避免残留店主/员工行 |
| 用户仅在商家端选 **仅商家身份（B）** | 客户端登录不受影响；客户端计划中的「整账号」仍独立可用 |
| 用户仅在商家端选 **整账号（A）** | 与客户端「整账号」一致；客户端须能处理「已从服务端删除」的登出态 |

**Stripe Connect（商家）**：按商家端计划 **§5.6** — 不在应用删号流程中强制断开 Stripe；商家仍可用 Stripe 官方渠道登录 Connect。**客户端删号**侧重 **Stripe Customer**、保存卡引用等（见 §6）。

## 4. 数据域与业务规则

以下 **§4.1–§4.3、§6** 已具备可开发结论；**§4.4** 仍为「建议默认」，T0 须 **逐项勾选为已确认** 并产出 **表/字段映射附录**（或 Confluence 链接），避免实现时遗漏边缘表。

**§4.2 与 §4.3 的执行顺序（避免重复处理）**  
同一张 `coupons`（及关联 gift 状态）上，**先按 Gift 规则分支**（§4.3：退回赠送人 / 对送出方失效并退款+通知），**其余未核销券**再按 §4.2 走「自动退款/失效」。实现时须在 Edge Function 内用**互斥条件**（例如先 `WHERE gift_...` 再 `WHERE NOT gift_...`）写清，避免一笔券既「退回」又「整单退款」双重入账。

**边界（Gift 退回失败 — 已确认）**  
- 若「退回至赠送人」**无法完成**（赠送人已删号、并发删号、脏数据、或 DB/RLS 无法将持有人写回赠送人等）：**Fallback = 按 §4.2 对该券执行与普通未核销券相同的自动退款/失效**，资金按既有规则退回，券作废。  
- **幂等（必须）**：同一 `coupon` / 关联订单若在**赠送人删号流程**或其它前置步骤中 **已退款或已失效**，接收人删号时 **不得再次退款**；实现上按 coupon id、退款状态或 Stripe 侧幂等键 **先读后写**。  
- 「失效并退款」「退回」须与现有 **退款 Edge / Stripe** 路径复用；**整账号删除宜异步队列或分阶段**（先禁止新支付 → 处理券 → 最后删 auth），避免单次 Edge 超时。

### 4.1 订单与支付

- **订单 `orders` / `order_items`**：匿名化 `user_id` 或保留不可识别引用；满足 **§5 匿名化保留**。  
- **已选方案**：保留**不可识别引用**（订单主键、金额、税等保留；`user_id` 置空或替换为统一占位匿名 ID，与 RLS/报表约定一致即可）。
- **退款 / 税务**：与现有 `tax_amount`、退款展示逻辑兼容（仅展示匿名化后的记录）。

### 4.2 优惠券与券状态（非 Gift 或 Gift 处理后的剩余）

- **未核销券（非 §4.3 已覆盖的 Gift 分支）**：**自动触发与业务一致的退款/失效**（与现有退款/订单状态机一致，禁止仅改 UI 不记账）。
- **已核销 / 已退款**：通常仅匿名化关联用户字段。

### 4.3 Gift（赠送券）

- **送出的 gift**：若未核销且仍有效 → **失效并退款**（走既有退款管道），并**通知接收人**（站内/邮件以现有能力为准）。  
- **收到的 gift**：若未核销且仍有效 → **退回至赠送人**（更新持有人/归属字段，与现有 `send-gift` / `recall-gift` 数据模型对齐），并**通知赠送人**。  
- **注意**：`COMPLETED.md` 中 **Gift 相关文件** 可能受保护 — 若改 `coupons_provider` / `coupon_screen` / Edge Functions，须单独确认授权；**优先在 Edge Function + SQL 完成数据迁移**，减少对受保护 Dart 文件的依赖。

**Gift 退回失败时的 Fallback（与上文「边界」一致，已锁定）**  
顺序上仍应先尝试 **退回赠送人**；仅当赠送人不可用或写回失败时，再进入 **§4.2 等价处理**。常态顺序删号下较少触发本 Fallback；主要用于并发、部分失败重试与异常数据。

### 4.4 其他客户端数据（grep 与 schema 逐项核对）

| 域 | 说明 | 建议默认（T0 可勾选为「已确认」） |
|----|------|----------------------------------|
| `saved_deals` / 收藏 | 删除或匿名化 | **硬删除行**（无财务链）较简单 |
| `reviews` | 匿名化作者或按政策保留 | **匿名化展示名 / user_id**，保留评价正文以利商户 |
| `friends` / `chat` / 会话 | 会话成员删除后的展示（占位用户 vs 级联） | **占位「Deleted user」+ 禁发消息**；避免删会话导致对端消息全丢（除非法务要求级联） |
| `referral` / `users.referral_code` | 解除上下级或匿名化 | **清空 `referred_by` 指向** + 匿名化或废弃 `referral_code` |
| `push` / `device` token | 删除订阅与 token | **删除** |
| `marketing_opt_in` / 邮件偏好 | 随用户行策略处理 | 随 `users` 匿名化或删行一并处理 |

## 5. 已对齐的跨端结论（来自商家端计划 §5）

| 事项 | 结论 |
|------|------|
| 订单/财务 | **匿名化保留** |
| 整账号 vs 仅商家 | 商家端 **A+B**；客户端仅 **整账号 `full`**（仅商家身份仅在商家端选 B） |
| Stripe Connect（商家侧） | **不**强制断开；商家可继续用 **Stripe 官方登录** Connect |
| 并行交付 | 与商家端 **同一阶段**；本文档为客户端专用任务拆解 |

## 6. Stripe（客户端侧重）— 推荐方案（与现有业务一致）

**目标**：删号后不在 Stripe 侧留下「仍绑定已删除 App 用户」的活跃支付方式；**不**影响商家 Connect（见商家端 §5.6）。

**推荐顺序（在 Edge Function / 服务端执行）**

1. **若存在进行中的 PaymentIntent / 未结清争议**：先按现有规则 **阻断删号** 或 **排队异步**（须在 API 返回明确错误码，App 展示英文原因）。  
2. **列出并 detach 所有 PaymentMethods**（`stripe_customer_id` 已知时）：调用 Stripe API 解除卡绑定，避免孤儿卡挂在 Customer 上。  
3. **删除 Stripe Customer**（`customers.delete`）：消费者侧常用做法；删除后历史 Charge 仍可在 Stripe Dashboard 查询，但 App 内不再持有 `cus_xxx`。若 Stripe 返回「有订阅/分期」等不可删，则降级为 **仅 detach PM + 将 Customer metadata 标记为 `deleted_at`** 并记录告警。  
4. **清空 `public.users`（或等价）中的 `stripe_customer_id` / 支付相关字段**，再执行 **auth 用户删除**（顺序避免 RLS 读旧 ID 失败）。

**与 §5.6 的关系**：商家 **Connect** 不因本流程关闭；本段仅处理 **消费者 Customer + PaymentMethods**。

## 7. 技术方案概要

### 7.1 后端

- 与商家端 **同一 Edge Function**：消费者整账号路径 = `full` 中「用户级」段（orders/coupons/users/auth 等）。  
- **RLS / ON DELETE**：migration 列出所有 `REFERENCES users(id)` / `auth.users`，避免删用户事务失败。

### 7.2 客户端 App（`deal_joy`）

- **入口**：`Profile` → **Delete account**（具体子路径与 UI 稿一致）。  
- **流程**：说明后果（含商家端同一账号）→ 二次确认（可选密码重验）→ 调用 API → 成功 `signOut` → 登录页/欢迎页。  
- **可能涉及文件**（示例，以实际拆任务为准）：`profile_screen.dart`、auth 相关 provider/repository（**受保护须授权**）。

### 7.3 测试

- 覆盖：无券用户、有未核销券、有 gift 进出、有未完成订单、有保存卡；以及「仅删客户端账号但商家端 B 已选」的交叉场景（若保留双路径）。
