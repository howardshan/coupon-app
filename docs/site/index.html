Brand Multi-Store

Allows a brand admin to create a single Deal that applies across multiple store locations, with each store independently confirming participation and linking its own menu item.

How It Works

Brand Admin creates brand
→
Adds store locations
→
Creates a brand Deal
→
Platform approves
→
Stores confirm participation
→
Deal goes live per store

Store Statuses

active
Store confirmed. Deal is live and visible to users at this location.
pending_store_confirmation
Waiting for the store owner to accept or decline. 48-hour window.
declined
Store declined or auto-timed out. Deal not shown at this location.
removed
Store exited after going active, or was kicked out by brand admin.

Key Tables

Table / FieldPurpose
brandsBrand entity. One brand can have many store locations.
brand_adminsUsers with admin rights over a specific brand.
merchantsIndividual store location. Has a brand_id FK when part of a chain.
deal_applicable_storesCentral join table. One row per deal × store. Tracks per-store status, price snapshot, confirmation and removal metadata.
deals.store_confirmationsJSONB set by brand admin at creation. [{store_id, pre_confirmed}]. Drives the activation trigger.
ℹ️
Legacy field: deals.applicable_merchant_ids UUID[] is kept for backward compatibility. Do not remove.

Create Brand

A brand represents a chain or franchise umbrella. Creating a brand is the first step before adding store locations or creating brand-wide deals.

Who Can Create a Brand

Add Store

Add a physical store location to the brand. Each store is an independent merchant record linked to the brand via brand_id.

Two Ways to Add a Store

A. Invite an existing merchant

If the store is already registered on DealJoy as an independent merchant, the brand admin sends an invitation. The store owner accepts, and their merchants.brand_id is set to this brand.

B. Create a new store directly

Brand admin fills in the new store's details (name, address, phone, hours, location coordinates). A new merchants row is created with brand_id already set. A login credential is generated or sent to the store manager.

Required Store Fields

FieldDescription
nameStore location name (e.g. "Downtown Dallas")
addressFull street address
cityCity — used in search_deals_by_city() filtering
lat / lngCoordinates — used in search_deals_nearby() radius filtering
phoneContact number shown to users

After Adding

The store appears in the brand admin's store list. It can now be selected when creating a new brand deal. If the store has an existing user account, they will see the brand in their merchant app immediately.

⚠️
Coordinates matter: Stores without lat/lng will not appear in GPS-based deal searches. Always set coordinates when adding a store.

Delete Store

Remove a store location from the brand. This is a high-impact action with cascading effects on active deals and coupons.

What "Delete" Means

Deleting a store from the brand removes it from all brand-wide deals it was participating in. This is equivalent to kicking the store out of all active deals at once.

Cascade Effects

Affected areaWhat happens
deal_applicable_storesAll rows for this store with deal_scope = 'brand_multi_store' and status active or pending are updated to removed. removed_at is recorded.
Active dealsIf removing this store causes any deal's active store count to drop to 0, that deal is automatically set to is_active = false.
Existing couponsCoupons purchased before removed_at can still be redeemed at this store. See Redeem Deal.
Store-only dealsAny deals the store created independently are NOT affected.
merchants.brand_idSet to NULL. The store becomes an independent merchant again.

Soft Delete vs Hard Delete

Recommended: Soft remove (set brand_id = NULL)

The store's merchant record is kept. Historical orders, coupons, and reviews remain intact. The store simply loses its brand affiliation and is removed from future brand deals.

Hard delete (DELETE merchants row)

Cascades to delete ALL data for this store. Only use if the store was created in error and has no real orders.

🚨
Irreversible: Once a store is removed from the brand, all its deal_applicable_stores records are set to removed. These cannot be re-activated. A new deal creation is required to bring the store back into a deal.

Store Status

A brand admin can temporarily suspend a store location by setting merchants.is_active = false. A suspended store's active deals are not removed — they remain in deal_applicable_stores — but the store will not appear in user-facing search results.

Menu Management

Brand admins can view but not edit a store's menu items. Menu items are managed by the store owner themselves (menu_items table, scoped by merchant_id). This is intentional — each store controls its own pricing.

ℹ️
store_original_price snapshot: Even if a store later changes a menu item's price, the discount displayed to users remains calculated from the price at Accept time. Editing menu prices does not affect live deals.

Activation Trigger

When a platform admin approves the deal (deal_status → active), the PostgreSQL trigger on_deal_activated automatically inserts rows into deal_applicable_stores for each store in store_confirmations.

✅
No manual follow-up needed. Once the platform approves, the trigger handles all store record creation automatically.

Store Confirmation

When a deal is approved and a store is not pre-confirmed, the store owner goes through an Accept or Decline flow with a 48-hour window.

→

Accept

Store owner taps Accept → calls PATCH /merchant-deals/:id/store-confirm with action: "accept". The RPC accept_deal_store() runs:

Field setValue
statusactive

Decline

Store owner taps Decline → action: "decline" → RPC decline_deal_store() → status set to declined. Irreversible.

48-Hour Auto-Timeout

A pg_cron job runs every hour. Any pending_store_confirmation record older than 48 hours is automatically set to declined.

Menu Item Matching

The Store Deal Confirm page automatically searches the store's menu for an item matching the deal title by name. Two scenarios:

ScenarioUI shown
Menu item foundItem name, price, and calculated discount %. Accept button enabled.
Menu item not foundWarning banner + "Add to Menu" button (pre-fills item name). Accept requires adding item first.

Per-Store Discount Formula

Shown on user-facing deal card after Accept

((store_original_price − deal.discount_price) / store_original_price × 100).round()%

ℹ️
Adding to menu and tapping Accept are two separate steps. This prevents stores from going live without a proper menu entry in place.

User-Facing Display

On the deal detail page, users see:

  • "Available at X locations" — live count from deal_applicable_stores WHERE status = 'active'
  • A list of active store cards, each showing its address and per-store discount chip
  • Stores with pending, declined, or removed status are hidden from users

Search & Discovery

The deal appears in city and GPS search results only when at least one active store is within the search area. Both RPC functions (search_deals_by_city, search_deals_nearby) enforce this via an EXISTS subquery on deal_applicable_stores.

Cancel a Deal

Deals can be cancelled at the brand level (entire deal taken down) or at the store level (one location exits while others stay live).

Brand-Level Cancellation

Brand admin cancels the entire deal. Sets deals.is_active = false and deal_status = 'inactive'. The deal disappears from all search results and deal detail pages.

⚠️ Effect on sold coupons

Coupons already purchased continue to exist. Users can request a refund through the standard refund flow. The deal card in the user's coupon list will show as "Expired / Cancelled."

Store-Level Exit — Voluntary

A store owner can choose to leave a deal they are currently active in.

→
Status → removed, removed_at recorded
→
If last active store → deal auto-deactivated

Store-Level Kick — By Brand Admin

Brand admin removes a specific store from a specific deal, or removes a store from the brand entirely (which batch-removes from all deals).

ActionScopeWhat's updated
Kick from one dealSingle deal_applicable_stores rowstatus → removed, removed_by, removed_at
Remove store from brandAll brand_multi_store rows for this storeSame fields, bulk update

Coupon Redemption After Cancellation

ScenarioCoupon still redeemable at cancelled store?
Coupon purchased BEFORE removed_at✅ Yes — purchase predates the exit
Coupon purchased AFTER removed_at❌ No — should not be possible if deal was properly deactivated
Brand-level cancellation (is_active = false)❌ No new purchases possible; existing coupons refunded or honoured per refund policy
⚠️
Auto-deactivation: remove_deal_store() counts active stores after every removal. If the count hits 0, the deal is automatically deactivated — no manual step needed.

How the merchant scan function validates a coupon QR code at a specific store location, accounting for per-store participation status and exit timing.

Redemption Flow

→

→
Status + timestamp check
→
Allow or reject

Confirm → POST /merchant-scan/redeem

Decision Table

No record — ❌ Reject — not applicable at this location

The purchase timestamp used in comparisons is coupons.created_at — the exact moment the coupon row was inserted when the user completed checkout.

Edit as Clone

When a merchant edits a deal, the system clones it instead of updating in place. This preserves purchase history and triggers a fresh admin review.

How It Works

The merchant-deals Edge Function PATCH handler distinguishes two scenarios:

ScenarioBehavior
Stock-only change (total_stock, remaining_stock)Updates the existing deal row in place. No new deal is created.
Sort-order-only change (sort_order_only: true)Updates sort_order in place. Used by drag-reorder.
Any other field changeClones the deal: creates a new row with status = pending, copies deal_images and deal_applicable_stores, then sets the original deal to inactive.

Clone Process

Merchant submits edit
→
PATCH /merchant-deals
→
New deal row (pending)
→
Copy images & stores
→
Old deal → inactive
ℹ️
If the deal has option_groups, they are also cloned to the new deal. If the edit payload includes new option_groups, those replace the cloned groups.

Products Data Format

Deals store their included products in a structured string format that carries quantity and price data.

⚠️
Naming note: The DB column is still called dishes for backward compatibility. In all Dart code, the field has been renamed to products.

Format

The deals.dishes column is a text[] array. Each element follows one of two formats:

FormatExampleDescription
name"Smoked Brisket"Legacy format — name only, no price
name::qty::subtotal"Smoked Brisket::2::30"New format — includes quantity and subtotal

Parsing (User App)

DealModel._parseProducts() splits each string by ::. If 3 parts exist, it extracts (name, qty, subtotal). The deal detail page renders each line as:

Smoked Brisket     ×2   $30

Merchant App: Building the Data

In deal_create_page.dart, the _dishesData getter constructs the array from selected menu items:

_selectedMenuItems.map((s) {
  final name = s.menuItem.name;
  final qty = s.quantity;
  final subtotal = ((s.menuItem.price ?? 0) * qty).toStringAsFixed(0);
  return '$name::$qty::$subtotal';
}).toList();

Admin Activate Logic

The admin portal's Activate button now only appears for deals in pending status.

Before

The Activate button showed for any deal where !isActive. This meant merchants who voluntarily deactivated a deal would see an Activate option in the admin panel — unintended.

After

Condition changed to dealStatus === 'pending'. Only new or recently-edited deals awaiting admin review show the Activate button.

FileChange
deal-review-actions.tsxActivate button condition: !isActive → dealStatus === 'pending'
